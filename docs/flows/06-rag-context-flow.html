RAG combines retrieval (finding relevant information from your codebase) with generation (LLM response). Instead of relying solely on the LLM's training data, we retrieve actual code and include it in the context for accurate, grounded responses.

üîÑ RAG Pipeline Flow

1

üì• Query Analysis

Analyze the user's query to understand what context is needed. Extract keywords, file patterns, and intent.

query = "Fix the authentication bug in the login handler"
# Extract: "authentication", "bug", "login", "handler"
# Intent: debug, files: auth*, login*
2

üîç Retrieve Relevant Files

Search the codebase for files matching the query. Use pattern matching, file names, and content search to find relevant code.

    "src/auth/login_handler.py",    # Name match
    "src/auth/authentication.py",   # Keyword match
    "src/middleware/auth.py",       # Related
    "tests/test_auth.py"            # Tests
]
3

üìä Prioritize & Filter

Score files by relevance and importance. Fill the context window while staying within token budget.

priorities = {

    "test_auth.py": 0.60,       # Tests
}
# Fill 32K budget with highest priority first
4

‚ú® Generate with Context

Send the query + retrieved context to the LLM. The response is grounded in actual code, not hallucinated.

{file_contents}

User query: {query}

Analyze the code and fix the bug.
"""
response = llm.generate(prompt)

üß† Smart Context Window Features

üéØ File Prioritization

Files are scored by extension (.py, .ts = high), name relevance, and recency. Most important files included first.

üìè Token Estimation

Estimates tokens using character count √ó 0.25. Ensures we never exceed the model's context window limit.

‚≠ê Auto-Include Important Files

Config files (package.json, pyproject.toml) and READMEs are auto-included for project context.

üîÑ Dynamic Refresh

Context is rebuilt for each request. Always uses current file state, never stale cached content.

üíæ Memory Integration

Recalls relevant past conversations and decisions to include in context for continuity.

üìé User Attachments

Files explicitly attached by user get highest priority. Always included in full.

üéØ Interview Key Points

Q: What is RAG and why is it important?

RAG = Retrieval-Augmented Generation. LLMs only know their training data. RAG allows us to inject real-time, project-specific context. The model's response is "grounded" in actual code rather than potentially hallucinated patterns. This is critical for code assistants that need to reference your actual codebase.

Q: How do you handle context window limits?

Smart prioritization and budget allocation. We estimate tokens (chars √ó 0.25), score files by relevance, and fill the context with highest-priority content first. If we exceed the budget, lower-priority files are truncated or excluded. We also reserve space for the response.

Q: What's the difference between RAG and fine-tuning?

RAG retrieves at runtime, fine-tuning trains into weights. RAG is dynamic - context changes with each query. Fine-tuning permanently encodes knowledge into the model. RAG is better for frequently changing codebases; fine-tuning is better for static domain knowledge.

Q: How do you decide which files to include?

Multi-factor scoring. We consider: (1) File extension priority (.py = 0.9), (2) Name relevance to query, (3) User attachments (always included), (4) Auto-important files (package.json), (5) Recency. Files are ranked and included until the budget is exhausted.
