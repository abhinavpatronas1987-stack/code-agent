@tool Decorator Pattern

from agno.tools.decorator import tool

@tool(name="read_file")
def read_file(
    file_path: str,
    start_line: Optional[int] = None,
    end_line: Optional[int] = None
) -> str:
    """
    Read contents of a file.

    Args:
        file_path: Path to the file to read
        start_line: Optional starting line
        end_line: Optional ending line

    Returns:
        File contents as string
    """
    # Implementation here
    path = Path(file_path)
    return path.read_text()

üéØ Interview Key Points

Q: How does the LLM know which tool to use?

Each tool has a detailed docstring that becomes its description in the schema. The LLM matches user intent to tool descriptions through its training. "Read config.py" maps to read_file because the description says "Read contents of a file."

Q: What is the @tool decorator pattern?

It's a Python decorator that transforms functions into LLM-callable tools. It extracts the function signature (parameter names, types) and docstring to generate a JSON schema automatically. This schema is sent to the LLM.

Q: How do you handle tool errors?

Tool execution is wrapped in try/catch. Errors are formatted as strings and returned to the LLM, which can then reason about the error and try a different approach. The error becomes part of the conversation.

Q: Why organize tools into categories?

Organization aids maintainability and specialized agents. For example, CodeReviewer only gets read/search tools (not write), reducing risk of unintended changes. It's the principle of least privilege.
